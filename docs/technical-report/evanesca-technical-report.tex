\noindent{\Large\bfseries Evanesca Technical Report v0.1}

\vspace{0.85em}
\noindent{\normalsize EVANESCA PROJECT}

\vspace{1.65em}

\noindent\textbf{Scope.}
\sys is a research prototype and open-source framework for reconstructing
decentralized-finance (DeFi) transactions as Semantic Financial Graphs (SFGs)
and auditing those graphs with transparent financial constraints. This
technical report reorganizes the paper-era Evanesca work as a public
whitepaper-style artifact. It explains the measurement problem, threat model,
semantic recovery pipeline, graph abstraction, constraint methodology,
evaluation snapshot, public examples, and release limitations.

\vspace{0.4em}
\noindent\textbf{Position.}
The framing is deliberately narrower than a production exploit detector. \sys
surfaces constraint-backed candidate events and replayable evidence traces for
analyst review; it does not infer malicious intent from traces alone. Using the
IMC-version measurement snapshot, \sys analyzes \numtransactions Ethereum
transactions from 2020--2024, reconstructs \numincidents confirmed incidents
across five chains, and flags \numflagged candidate instances, including
\numbaseline routine arbitrage baselines and \numnonbaseline non-baseline
instances. Source-code investigation of the non-baseline set identifies
\numdefects instances in two under-studied defect families:
reward-distribution manipulation and derivative-pricing/yield-token deviation.

\section{Introduction}
\label{sec:introduction}

DeFi incidents are usually explained as isolated exploits: one protocol, one
transaction, one postmortem. Yet many incidents share a common structure at the
level of financial actions. A bZx-style price manipulation, a Harvest-style
exchange-rate manipulation, a reward-distribution abuse, and a derivative-token
pricing error may involve different contracts and event schemas, but each can
be reconstructed as an ordered sequence of swaps, deposits, withdrawals,
borrows, repayments, and profit-taking steps. The repeated structure suggests
that value extraction can be measured systematically rather than investigated
only case by case.

This observation creates a measurement challenge. Existing systems address
important parts of the DeFi security problem: DeFiRanger and BlockEye detect
known attack patterns or price-manipulation behavior~\cite{wu2021defiranger,wang2021blockeye};
DeFort and DeFiScope focus on price-manipulation analysis with specialized
logic~\cite{defort2024,zhong2025defiscope}; and operational systems such as
Forta and OpenZeppelin Defender provide deployment-facing monitoring and
alerting~\cite{forta2023whitepaper,ozdefender}. These systems are useful, but
they often bind analysis to a deployment-specific monitor, predefined attack
template, or specific attack family. They also tend to emit alerts rather than
the per-step financial computation needed for independent replay and triage.

\sys takes a different position. It is a measurement and forensic
reconstruction framework. It recovers protocol-specific traces into uniform
financial actions, constructs a Semantic Financial Graph, evaluates
service-class financial constraints, and exports a replayable evidence trace.
Each match is a \emph{candidate event}, not a verdict that the transaction was
malicious or harmful. This distinction matters because DeFi value extraction
does not divide neatly into attacks and normal behavior. MEV and routine
arbitrage can move value without a software defect~\cite{flashbots,weintraub2022flash,mevdef},
while flash-loan and oracle incidents can reuse similar swap and borrow
primitives to exploit a protocol-local economic assumption~\cite{qin2021attacking,zhou2023sok}.

\paragraph{Report scope.}
This report is not a resubmission of the IMC short paper and is not meant to
replace a future conference manuscript. It is a public technical report for the
open-source release. The empirical numbers in the evaluation section are the
IMC-version measurement snapshot unless explicitly labeled as validation or
release evidence. Figures are reused from IMC, DeFi 26, and arXiv-era Evanesca
drafts when they explain the system or a case study; non-IMC scale-out claims
from older drafts are intentionally excluded.

\paragraph{Contributions.}
The public release centers on five concrete contributions.
\begin{itemize}
  \item \textbf{A protocol-independent forensic abstraction.} \sys maps
  heterogeneous DeFi traces into typed financial actions and then into SFGs.
  The novelty claim is not that graphs are new; it is that action recovery,
  graph construction, constraint evaluation, and evidence export are tied
  together as a reproducible measurement workflow.
  \item \textbf{Transparent constraint-based measurement.} \sys uses
  \numconstraints constraints in \numdsllines lines of DSL to encode
  service-class financial assumptions rather than incident-specific
  signatures.
  \item \textbf{A scoped empirical snapshot.} The IMC-version evaluation
  analyzes \numtransactions Ethereum transactions and separates routine
  arbitrage from non-baseline candidate families.
  \item \textbf{Replayable evidence artifacts.} The public release exports
  JSON evidence traces for bZx and Harvest case studies and includes manifests
  for larger public hash-set artifacts.
  \item \textbf{An open-source release path.} The release includes source code,
  documentation, public artifact validation, example datasets, and explicit
  limitations so that future work can extend the measurement pipeline without
  overstating the current prototype.
\end{itemize}

\section{Definitions and Threat Model}
\label{sec:threat}

\subsection{Terminology}

\sys separates measurement signals from final security judgments.
\begin{description}[leftmargin=1.3em,style=nextline]
  \item[Candidate event.]
  A transaction or trace for which at least one financial constraint fires. A
  candidate requires review; it is not automatically an exploit.
  \item[Benign arbitrage baseline.]
  Value extraction that resolves price discrepancies across markets without
  evidence of violating a protocol-local accounting, oracle, collateral,
  reward, bridge, or derivative-pricing rule.
  \item[Harmful extraction.]
  Abnormal value transfer caused by exploiting or violating a protocol-local
  economic rule, such as a collateral relation, exchange-rate calculation,
  reward-allocation rule, bridge backing invariant, or oracle assumption.
  \item[Confirmed incident.]
  A publicly documented event with replayable on-chain transaction evidence
  and an external narrative. Confirmed incidents are used as validation anchors,
  not as a complete population sample.
\end{description}

\subsection{Adversary Model}

The actor may use one or more externally owned accounts or contracts, compose
multiple DeFi protocols within one transaction, borrow temporary capital via a
flash loan, manipulate an AMM or oracle-dependent state, trigger reward logic,
and unwind positions before transaction completion. The actor may be a routine
arbitrageur, MEV searcher, liquidator, or attacker. \sys does not require
private mempool access, block-builder control, or sandwich capability as a core
assumption; those behaviors can be represented when visible in traces, but the
primary reconstruction unit is the transaction-local financial action sequence.

\subsection{Loss Subjects}

The loss subject depends on the service class. \sys distinguishes protocol
reserves, liquidity providers, lending depositors, reward-pool participants,
derivative-token holders, bridge backing reserves, and traders receiving worse
execution. Some profitable transactions have no direct harmed protocol and are
best described as market-efficiency arbitrage. This is why routine arbitrage is
kept as an explicit baseline rather than collapsed into false positives.

\begin{findingbox}
\noindent\textbf{Design position.} \sys measures replayable financial-action
patterns. It does not infer real-world identity, intent, or legal culpability.
The output is evidence for analysts, not an automated accusation.
\end{findingbox}

\section{Measurement Questions}
\label{sec:questions}

The paper-era evaluation was organized around three research questions that
remain useful for the public report.
\begin{itemize}
  \item \textbf{RQ1: Incident reconstruction.} Can general-purpose constraints
  reconstruct known DeFi incidents without per-incident logic?
  \item \textbf{RQ2: Pattern landscape.} What candidate value-extraction
  patterns emerge at scale once benign arbitrage is separated from
  non-baseline behavior?
  \item \textbf{RQ3: Defect validation.} Do non-baseline candidates correspond
  to concrete implementation defects under source-code and event-log review?
\end{itemize}

The public release adds a fourth practical question: can the system expose
enough code, data, and documentation for another researcher to inspect how a
candidate was reconstructed? This report is structured around that
reproducibility requirement.

\section{System Overview}
\label{sec:overview}

Figure~\ref{fig:pipeline} shows the pipeline. Each stage has a defined input
and output, so failures can be localized: semantic recovery may miss an
unsupported protocol, SFG construction may lack token metadata, constraint
evaluation may fire too broadly, and evidence export may expose an incomplete
diagnostic.

  \centering
  \includegraphics[width=0.98\linewidth]{new_overview.pdf}
  \caption{\sys pipeline: semantic recovery, SFG construction, constraint
  evaluation, and evidence export. The figure is reused from paper-era
  Evanesca drafts as system documentation.}
  \label{fig:pipeline}

\paragraph{Semantic recovery.}
Protocol-specific adapters decode traces and logs into uniform financial
actions. The public implementation covers common DEX, lending, flash-loan, and
bridge families, but adapter coverage remains incomplete. The adapter boundary
is deliberate: protocol-specific decoding belongs in semantic recovery, while
constraints should operate over service-class financial actions.

\paragraph{SFG construction.}
Recovered actions are assembled into a directed graph \(G=(V,E)\). Vertices
represent accounts, pools, markets, protocol contracts, and token entities.
Edges represent typed financial actions and carry token amounts, token
addresses/symbols when known, USD-normalized values, protocol context,
ordering, and raw evidence pointers.

\paragraph{Constraint evaluation.}
Constraints encode financial assumptions common to a service class, such as
value preservation across swaps, collateral sufficiency, exchange-rate
consistency, oracle sanity, reward-allocation stability, and bridge backing.
Constraints are measurement hypotheses. A match is a candidate event.

\paragraph{Evidence export.}
The public CLI exports JSON containing transaction identity, a summary, graph
nodes and edges, fired constraints, evidence steps, and diagnostics. The schema
is documented in \path{docs/public-api/analysis-result.schema.md}; examples
live in \path{docs/example-datasets/}.

\section{Semantic Financial Graph}
\label{sec:sfg}

The SFG abstraction is designed to bridge the semantic gap between raw token
transfers and financial interpretation. Transfer-only representations are often
insufficient for DeFi incidents because profit can depend on collateral,
share-price accounting, reward entitlement, virtual reserves, oracle reads, or
derivative exchange rates. SFG edges therefore represent typed financial
actions rather than only ERC-20 transfer events.

\paragraph{Difference from cash-flow trees.}
\defiranger constructs cash-flow trees from raw transfer and balance-change
patterns~\cite{wu2021defiranger}. That is a strong fit for a subset of price
manipulation attacks, but transfer-level templates struggle when the important
state is a share price, collateral relation, reward trigger, or bridge backing
rule. \sys does not claim that SFGs are categorically better for every task.
Rather, SFGs are the representation Evanesca needs to make cross-service
constraints and per-step PnL attribution replayable.

\section{Constraint Methodology}
\label{sec:constraints}

For an edge \(e\) with nonzero USD input value, \sys commonly computes:
\[
  \rho(e) =
  \frac{\mathrm{valueUSD}(\mathrm{out}(e))}
       {\mathrm{valueUSD}(\mathrm{in}(e))}.
\]
The paper-era implementation used \(\alpha=0.05\) for DEX swap edges and
\(\beta=0.02\) for lending or oracle-mediated stablecoin-derivative edges. The
DEX threshold is above reported routine unexpected slippage on Uniswap
\cite{zhou2021high}; the stablecoin-derivative threshold is tighter because
oracle studies report smaller normal movements for major stable assets
\cite{liu2021firstlook}. These thresholds are measurement parameters, not
universal constants. A stronger future paper should include sensitivity
analysis over \(\alpha\), \(\beta\), market regime, and protocol class.

\paragraph{DSL shape.}
Each constraint declares when it applies, which intermediate variables it
derives, and which violation predicate fires. A simplified example is:
\begin{verbatim}
constraint PRICE_MANIPULATION {
  when: edge.Type == "DEX" && edge.Action == "Swap"
  condition: {
    price_ratio: edge.totalOutUSD / edge.totalInUSD
  }
  violation:
    edge.totalInUSD > 0 && price_ratio > 1.05
}
\end{verbatim}
Production constraints add accounting guards and service-specific fields when
available, but the key property is auditable replay: the predicate, derived
values, and evidence edge are exported together.

\section{Datasets and Scope}
\label{sec:datasets}

\subsection{Scale-Out Dataset}

The IMC-version scale-out evaluation covers \numtransactions Ethereum
transactions from January 2020 through December 2024 that match Evanesca's
supported adapter set. The remaining chains in the paper-era evaluation
(BSC, Arbitrum, Avalanche, and Optimism) contribute confirmed-incident
validation examples, not RQ2/RQ3 population prevalence estimates. Counts in
this report should therefore be read as Ethereum scale-out results unless
explicitly stated otherwise.

\subsection{Confirmed-Incident Validation Set}

The \numincidents confirmed incidents are publicly documented events with
available transaction traces across the supported chains. A future polished
paper should publish a full inclusion/exclusion protocol. The intended
inclusion criteria are:
\begin{itemize}
  \item public postmortem, incident report, or external narrative exists;
  \item exploit or representative transaction hash is available;
  \item on-chain trace/log data is accessible;
  \item economic loss or value transfer is attributable to on-chain behavior;
  \item the target chain and protocol family are supported by the collector.
\end{itemize}
Excluded categories include off-chain compromises without meaningful DeFi
interaction, rug pulls without reconstructable financial actions, unsupported
non-EVM chains, bridge-validator compromises without transaction-local
financial reconstruction, and unavailable traces.

\subsection{Public Example Datasets}

The public release includes a synthetic schema example and two replayed
transaction examples:
\begin{itemize}
  \item \path{docs/example-datasets/bzx-analysis-result.json};
  \item \path{docs/example-datasets/harvest-analysis-result.json}.
\end{itemize}
These files are evidence-format examples, not a complete benchmark. The bZx
example is byte-stable in the current public fallback environment. The Harvest
example is schema-stable but provider-sensitive: public fallback RPC can miss
Curve token metadata that configured archive/API providers recover, changing
profit-attribution diagnostics while preserving the graph/evidence shape.

value-extracting step, traces a profit path to the beneficiary, and fires at

\subsection{RQ3: Implementation-Defect Validation}

The paper-era source-code and event-log review identifies \numdefects
instances in two defect families. The \numyield yield-bearing token deviations
map to derivative-pricing or exchange-rate errors. The \numreward
reward-distribution instances map to reward-allocation bias where payout logic
uses momentary balances rather than sustained exposure.

\begin{figure}[t]
  \centering
  \includegraphics[width=0.98\linewidth]{case_tend.pdf}
  \caption{Representative reward-distribution manipulation pattern. A
  flash-loan-funded path triggers reward payout and immediately unwinds.}
  \label{fig:tend}
\end{figure}

\paragraph{Derivative pricing and yield-token deviation.}
These cases involve cross-protocol deposit/withdraw or redemption paths where a
receipt token, vault share, or derivative asset is priced inconsistently across
services. The extraction is not visible as a single suspicious transfer; it
appears when the SFG links acquisition, conversion, redemption, and unwind
steps.

\paragraph{Reward-distribution bias.}
These cases use temporary liquidity or momentary balances to capture rewards
without sustained holding. The public report keeps the numeric finding from the
IMC snapshot but avoids claiming current exploitability or active disclosure
status beyond what the public artifacts can substantiate.

\begin{findingbox}
\noindent\textbf{Finding (RQ3).} Of the \numnonbaseline non-baseline
instances, \numdefects correspond to concrete implementation-defect families
under source-code/event-log review. The remaining non-baseline candidates
include price manipulation and unresolved long-tail cases, so the public
release should not describe every candidate as a vulnerability.
\end{findingbox}

\paragraph{Claim policy.}
Every public empirical claim should identify the code revision, input hashes or
dataset manifest, command line, configuration, external API/RPC requirements,
and expected output. Claims that require private credentials or unpublished
datasets must be marked as unreproduced in the public release.

\section{Limitations}
\label{sec:limitations}

\sys remains a research prototype. The main limitations are:
\begin{itemize}
  \item \textbf{Adapter coverage.} Unsupported protocols can produce false
  negatives or incomplete evidence traces.
  \item \textbf{Trace and RPC dependence.} Public fallback RPC may omit token
  metadata or historical state needed for richer diagnostics.
  \item \textbf{USD normalization sensitivity.} Historical price sources can
  affect profit diagnostics, especially for thinly traded or derivative tokens.
  \item \textbf{Threshold sensitivity.} \(\alpha\) and \(\beta\) are defended
  measurement parameters, not universal constants.
  \item \textbf{Manual labeling.} The public release does not yet provide a
  complete precision/recall evaluation over the \numflagged candidate set.
  \item \textbf{Build status.} The source tree has a public CLI smoke path and
  artifact validation, but repository-wide TypeScript build cleanup remains a
  release gate.
\end{itemize}

\section{Ethics and Disclosure}
\label{sec:ethics}

The release should avoid publishing exploit recipes beyond already public
transaction traces and high-level financial-action explanations. The public
examples use historical transactions and focus on reconstruction, not live
attack enablement. Candidate labels must be phrased conservatively: flagged
transactions are evidence for review, not accusations. Any future disclosure of
newly active defects should follow coordinated disclosure norms before public
release.

\section{Future Work}
\label{sec:future}

A future conference submission remains optional. If pursued, it should not
reuse this technical report as-is; it should add the missing evaluation rigor:
\begin{itemize}
  \item explicit inclusion/exclusion criteria for the \numincidents incident set;
  \item semantic-recovery correctness evaluation with sampled manual labels;
  \item candidate classification protocol with multiple annotators;
  \item precision/recall or top-\(k\) triage evaluation;
  \item threshold sensitivity analysis for \(\alpha\) and \(\beta\);
  \item case-by-case comparison against prior systems and ensembles;
  \item held-out protocol evaluation to quantify adapter generality.
\end{itemize}

\section{Conclusion}
\label{sec:conclusion}

\sys exposes DeFi transactions as replayable financial-action evidence rather
than opaque alerts. This report reframes the paper-era prototype as a public
open-source technical artifact: a source tree, SFG/evidence schema, constraint
methodology, IMC-scoped measurement snapshot, replayed examples, and
reproducibility path. The right success criterion for the release is not
whether every candidate is automatically labeled, but whether another
researcher can inspect how a candidate was reconstructed and decide what claim
is justified.
